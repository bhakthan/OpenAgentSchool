A Closer Look — How a Concept Page Works

Each of the 85+ concepts follows the same pattern. Here's how ConceptsHub.tsx defines a concept.

interface ConceptInfo {
  id: string
  title: string
  description: string
  level: 'fundamentals' | 'architecture' | 'implementation' | 'advanced' | 'applied'
  icon: React.ReactNode
  color: string
  estimatedTime: string
  prerequisites: string[]
  component?: React.ComponentType
}
TypeScript interface "This is the blueprint for every concept in the app."
id "A unique kebab-case name like 'agent-architecture'"
title "The display name shown on the card"
description "A short description (under 140 characters)"
level "Which learning tier it belongs to — from basics to applied"
icon "The icon component shown on the concept card"
color "Tailwind CSS classes for the card's color scheme"
estimatedTime "How long the learner should expect to spend"
prerequisites "Which other concepts should be completed first"
React.ComponentType "The React component that renders the full concept page"

Every concept card you see on the hub is generated from an object matching this shape — data in, UI out.

The Big Takeaway

💡 Key Insight

Notice how each concept page is its own file — AgentArchitectureConcept.tsx, MCPConcept.tsx, etc. This pattern is called separation of concerns.

Each file has one job. If something breaks in the MCP page, you know exactly where to look. When talking to an AI coding agent, you'd say "edit the MCPConcept component" and it knows precisely which file to change.

Lazy Loading — Load Only What You Need

With 85+ concept pages, loading them all upfront would be brutal. Lazy loading fixes this. 

const ConceptsExplorer = lazy(() => import('./components/concepts/ConceptsExplorer'));
const PatternExplorer = lazy(() => import('./components/patterns/PatternExplorer'));
const StudyMode = lazy(() => import('./components/study-mode/StudyMode'));
const TreeVisualizationPage = lazy(() => import('./components/pages/TreeVisualizationPage'));

Don't load ConceptsExplorer code until someone actually visits that page

Same for patterns — only fetch the code when needed

Study mode is heavy (AI integration) — definitely lazy-load it

The D3 tree visualization is huge — lazy loading keeps initial load fast

💡
Key Insight

Lazy loading is one of the most impactful performance patterns in modern web apps. Instead of shipping ALL 85 concept pages to every visitor, only the one they click on gets downloaded. It's like a streaming service that buffers only the scene you're watching, not the entire movie.

Talking to the Backend — The API Client

When the app needs data from the server, it uses an Axios-based API client with automatic auth. 

class APIClient {
  constructor() {
    this.client = axios.create({
      baseURL: `${API_BASE_URL}/api/v1`,
      headers: { 'Content-Type': 'application/json' },
      timeout: 30000,
    });

    this.client.interceptors.request.use((config) => {
      const token = this.getToken();
      if (token) {
        config.headers.Authorization = `Bearer ${token}`;
      }
      return config;
    });
  }
}
The APIClient class — a single object that handles all backend communication
When the class is created, set up the HTTP client
Create an Axios instance pointed at the backend server
Base URL: all requests go to the backend's /api/v1/ path
Tell the server we're sending JSON data
Give up after 30 seconds if the server doesn't respond
Before EVERY request is sent out, run this check first
Look for a saved login token in the browser
If we found a token...
Attach it as a Bearer token in the request header
Send the request on its way

An interceptor attaches the JWT as a Bearer token to every outgoing request automatically.

React Query — Smart Data Fetching

The app doesn't just fetch data — it caches it, deduplicates requests, and retries on failure. This is all handled by React Query (TanStack Query), wrapped in QueryProvider at the root.

🔄
Automatic Caching

Fetch data once, reuse it everywhere — no redundant API calls

🔁
Background Refetch

Stale data gets refreshed automatically when the user returns to a tab

⚡
Deduplication

If 3 components request the same data, only 1 API call is made

🔧
Error & Retry

Failed requests are retried automatically with exponential backoff

Callout + Key Insight

💡
Key Insight

The Provider pattern is one of the most important architectural decisions in this codebase. By nesting providers, each component can access theme, auth, settings, and feature flags without prop drilling — passing data through 10 levels of components manually. If you're building your own React app and an AI agent suggests "just pass it as a prop," ask yourself: would a Provider be cleaner?

The Clever Tricks

Production-grade patterns that separate a weekend project from a real product.

The Metaphor — A Swiss Army Knife

🔧

A Swiss Army knife doesn't expose all its tools at once. You pull out the blade when you need it, the screwdriver when you need that. This codebase works the same way — code splitting, feature flags, and access control ensure users only load and see what they need.

Pattern 1 — Code Splitting & Lazy Loading

With 85+ concept pages and 50+ visualization components, the initial JavaScript bundle could be massive. Code splitting breaks it into chunks that load on demand.

📦

React.lazy()

Each concept page is wrapped in lazy() — code downloads only when visited.

⏳

Suspense Fallback

While a chunk loads, a spinning loader appears — no blank screens.

🗂️

Dynamic Imports

import('./MCPConcept') tells Vite to create a separate .js file for that component.

🧮
By the Numbers

Without code splitting, every visitor would download code for 85 concepts + 70 patterns + 50 visualizations. With lazy loading, the initial bundle is just the shell — each page adds only 10-50KB on demand.

Pattern 2 — Feature Flags & Access Control

Not every feature is available to every user. The app uses two gatekeeping patterns.

<Route path="/phase1-lab" element={
  <AuthGuard>
    <FeatureFlagGate flag="phase1-lab" title="Phase 1 Lab">
      <Phase1LabPage />
    </FeatureFlagGate>
  </AuthGuard>
} />

<Route path="/adoption-playbook" element={
  <ModuleAccessGate module="adoption-playbook" title="Adoption Playbook">
    <AgenticAdoptionPlaybook />
  </ModuleAccessGate>
} />

Route for /phase1-lab — but with guards:

First gate: Are you logged in? If not, redirect to login.

Second gate: Is the 'phase1-lab' feature flag turned on? If not, show a "coming soon" message.

Only if BOTH gates pass, render the actual page.

 

 

 

Different gatekeeping pattern for the Adoption Playbook:

ModuleAccessGate checks your learning profile tier — is this content unlocked for your level?

 

 

Access control ensures users see only what they're authorized to see — whether by login status, feature flag, or learning tier.

Pattern 4 — Graceful Error Handling

The app uses defensive coding to prevent crashes from propagating.

🛡️

ResizeObserver Suppression

ReactFlow diagrams can trigger harmless but noisy errors — the app catches and silences them.

🔄

Layout Shift Recovery

If a component causes a layout shift, the app detects it via PerformanceObserver and self-heals.

🌐

Offline Banner

When connectivity drops, a banner appears and features degrade gracefully.

💡
Key Insight

Error handling isn't glamorous, but it's what separates production apps from demos. App.tsx has 100+ lines dedicated purely to catching, suppressing, and recovering from errors. When you're building with AI tools, always ask: "What happens when this fails?" The answer should never be "it crashes."

How It All Flows — End to End

Let's trace what happens when a logged-in user takes a quiz.

1 User clicks 'Knowledge Quiz'

React Router matches /quiz → lazy-loads QuizSection component

2 QuizSection mounts

React Query fires a GET to /api/v1/quiz/questions/{category} via the APIClient

3 APIClient adds auth

The request interceptor attaches the JWT Bearer token automatically

4 Core API responds

The backend fetches questions from DuckDB, returns JSON with questions and options

5 React Query caches it

The response is cached — revisiting the quiz page won't trigger another API call

6 User submits answers

A POST to /api/v1/quiz/submit sends answers. Backend scores them and returns results.
